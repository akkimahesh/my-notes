namespace
================
namespace is a isolated space inside the cluster, we can configure project related resources into it. we can easily manage multiple projects without any confusion.

kubectl get ns --> get available namespaces

pod
=====
pod is smallest deployable unit in kubernetes. a pod can have multiple containers in it. containers inside pod share same network and storage.

points:
#inside pod containers share same network and storage.

# Display Pod details, including Pod IP and Node information (default namespace)
kubectl get pod <pod-name> -o wide

# Display Pod details, including Pod IP and Node information (specific namespace)
kubectl get pod <pod-name> -n <namespace-name> -o wide

# Login to a specific container in a multi-container Pod
kubectl exec -it <pod-name> -c <container-name> -- bash (default namespace)
kubectl exec -it multi-container -n roboshop -c nginx -- bash (custom namespace)

# Example
kubectl exec -it multi-container -c nginx -- bash

# Login to a Pod that contains only a single container (specific namespace)
kubectl exec -it <pod-name> -n <namespace-name> -- bash

# Example
kubectl exec -it my-pod -n dev -- bash

Why inside pod have multiple containers?
Because one  main container is busy with serving content to users...another container takes logs of that container and push it to ELK.

warn:
don't add multiple containers in single pod if they are independent. [like frontend, database]


labels:
Labels are used to provide additional metadata for Kubernetes resources. They help identify, organize, and group resources, and can be used as selectors to associate resources with one another. For example, a Pod can use label selectors to reference and interact with a ConfigMap or other related resources.

annotations:
annotations are also like label, but here we can attach external resources out of kuberbernets like, Jenkins api address

resources:
using resources field we can limit the resources utilizations inside the node. request and limit resources.

env:
env is a key:value data to store environment variables inside the pod. and we use whenever we want.

configmap:
configmap also like env but here we can store more key:value data which is not sensitive without disturbing pod manifest we can edit configmap if any changes to be occur. using configmap label or metadata we can attach to pod.

secrets:
secrets are used to store the sensitive information but it stores base64 format which is decodes easily. most of the time secrets used for api not for critical databases credentials, we store database credentials in secrets manager.

clusterIP:
clusterIP is only for internal Kubernetes cluster, not accessed by outside.

service(clusterIP) -> pod

NodePort:
NodePort opens random port to nodes within range 30000-32767. Because to prevent comflicting services inside cluster. 
Anyone who comes to Gate 30080 should be sent to Room 80

user->nodeport->clusterIp->pod(app)

LB:
LoadBalancer generates one DNS, instead of giving ip address to public we give loadbalancer DNS for security purpose, we can archieve this only in cloud providers.

user->LB->nodeport->clusterIp->pod(app)

replicaset:
if you want to create multiple pods. what you do..? we maintain separate yml file for each pod like nginx-a, nginx-2... this is not efficient way right..?
instead of that we use replicaset to archieve that, using replicaset we got desired number of pods, if any pod got terminated, or stopped with no reason it creats new same pod again.
the main replicaset job is to keep running desired number of pods. and saves the pod information for rollback.

deploymentset but commonly called as deployment:
deployment used to rollback to previous version if any abnormal behaviour of current version. Easy to shift another version of application.

scenario:
v1 is running later v2 is implemented easily update that, without any downtime. if v2 has some issues then rollout to v1 easily.

deployment->replicaset->pod

if you create deployment automatically you're creating replicaset and pod also.

deployment commands:

kubectl rollout history deployment/nginx-deployment -n roboshop 

kubectl rollout undo deployment/nginx-deployment --to-revision=4 -n roboshop

kubectl describe deployment nginx-deployment -n roboshop

#login into pod that created by Deployment
kubectl exec -it nginx-deployment-66cb7d8dd9-5fxjr -n roboshop -- sh
